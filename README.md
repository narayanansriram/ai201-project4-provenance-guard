# Provenance Guard

A backend classification system that any creative sharing platform could plug into to determine whether submitted text was written by a human or generated by an AI tool. It classifies content, scores confidence, surfaces a transparency label, and handles creator appeals.

---

## Architecture Overview

A submitted piece of text travels through the following path:

```
POST /submit  { text, creator_id, content_type? }
        │
        ▼
   Validate input · generate content_id (UUID)
        │
        ├──────────────────┬──────────────────┐
        ▼                  ▼                  ▼
 detector_llm.py   detector_stylo.py   detector_bigram.py
 Groq LLM          sentence variance   bigram repetition
 (llama-3.3-70b)   TTR · punct/SVO     rate
 llm_score 0–1     stylo_score 0–1     bigram_score 0–1
        │                  │                  │
        └──────────────────┴──────────────────┘
                           │
                           ▼
                      scorer.py
          confidence = 0.45×llm + 0.30×stylo + 0.25×bigram
          voting layer → attribution (likely_ai / uncertain / likely_human)
                           │
                           ▼
                      labeler.py
               confidence → transparency label text
                           │
               ┌───────────┴───────────┐
               ▼                       ▼
          auditor.py              JSON response
       append to              { content_id, attribution,
       audit.jsonl               confidence, all scores,
                                 label, status }

POST /appeal  { content_id, creator_reasoning }
        │
        ▼
   storage.py  status → "under_review"
   auditor.py  append appeal entry
        │
        ▼
   JSON { appeal_received: true, status: "under_review" }
```

1. **`POST /submit`** receives `{ text, creator_id, content_type? }` and generates a `content_id` (UUID).
2. **`detector_llm.py`** sends the text to Groq (llama-3.3-70b-versatile) and returns a semantic AI-likelihood score (0.0–1.0). Uses a modified prompt when `content_type` is `image_description`.
3. **`detector_stylo.py`** computes sentence length variance, type-token ratio, and punctuation density in pure Python and returns a structural AI-likelihood score (0.0–1.0). For `image_description`, `detector_image_meta.py` substitutes an SVO pattern score for punctuation density.
4. **`detector_bigram.py`** computes bigram repetition rate — how often word pairs repeat — and returns a lexical/collocational AI-likelihood score (0.0–1.0).
5. **`scorer.py`** combines all three scores using ensemble weights (45% LLM, 30% stylometric, 25% bigram) plus a voting layer into a single `confidence` value and `attribution` string.
6. **`labeler.py`** maps the confidence score to one of three transparency label variants.
7. **`auditor.py`** appends a structured JSON entry to `logs/audit.jsonl` capturing all signals, scores, and status.
8. The endpoint returns the full record — `content_id`, `attribution`, `confidence`, `llm_score`, `stylo_score`, `bigram_score`, `votes_ai`, `label`, `status` — as JSON.

For appeals: `POST /appeal` receives `{ content_id, creator_reasoning }`, updates the stored record's status from `"classified"` to `"under_review"` in `storage.py`, and appends a separate appeal entry to the audit log. No automated re-classification occurs.

## Detection Signals

### Signal 1 — LLM-based classification (Groq)

**What it measures:** Semantic and stylistic coherence holistically. The model assesses whether the text reads like AI output — characteristic hedging phrases ("it is important to note that"), over-balanced structures, formulaic transitions, uniform register, and confident-but-hollow elaboration. This is a judgment no set of rules can fully replicate.

**Why this signal:** It captures what stylometrics cannot — the *meaning* of the text, not just its surface statistics. A human who writes in a formal register still uses idiosyncratic word choices, digresses, and contradicts themselves. An LLM doesn't.

**Output:** A single float 0.0 (clearly human) to 1.0 (clearly AI). The Groq prompt requests `{ "score": float, "reasoning": string }` so it can be parsed reliably.

**What it misses:** Heavily edited AI output; very short texts (< 80 words) where scores cluster near 0.5; ESL writers whose formal syntax mimics AI register; high-quality narrative AI writing that avoids all surface markers.

---

### Signal 2 — Stylometric heuristics (pure Python)

**What it measures:** Statistical structural properties that differ between human and AI writing. AI text is more uniform; human writing is more variable.

**Three sub-metrics (weights: 0.40 / 0.40 / 0.20):**

| Sub-metric | What it captures | AI pattern |
|---|---|---|
| Sentence length variance | Std dev of word counts across sentences | Low variance — narrow 12–20 word band |
| Type-token ratio (TTR) | Unique words / total words | Low TTR — repetitive vocabulary from training distribution |
| Punctuation density | (commas + semicolons + em-dashes) / word count × 100 | Above-baseline density |

**Why this signal:** Genuinely independent from Signal 1 — structural, not semantic. Two signals that capture different dimensions are more informative than two that both look at semantics.

**Output:** A single float 0.0–1.0 combined from the three sub-metrics.

**What it misses:** Formal human writing (academic papers, legal briefs) with low TTR and low sentence variance; poetry using intentional repetition (anaphora); very short texts (< 5 sentences) where variance estimates are unreliable.

Every `POST /submit` response includes `llm_score`, `stylo_score`, and `bigram_score` as individual fields alongside `confidence`, so callers can see exactly how each signal contributed to the combined result.

## Confidence Scoring

**Formula (3-signal ensemble):**
```
confidence = (0.45 × llm_score) + (0.30 × stylo_score) + (0.25 × bigram_score)
```

The LLM signal receives the most weight (45%) because it captures semantic patterns no statistical heuristic can. The two structural signals (stylometric + bigram) together carry 55%, counterbalancing potential LLM hallucination. The bigram signal is genuinely independent — it operates at the word-pair level, not the sentence or semantic level.

**Voting layer (attribution only, not used for confidence score):**
Each signal votes AI if its score > 0.50. Attribution is determined by:
- 3/3 votes AI → `likely_ai`
- 2/3 votes AI + confidence > 0.65 → `likely_ai`
- 2/3 votes AI + confidence ≤ 0.65 → `uncertain`
- 0–1 votes AI → falls back to confidence thresholds

This prevents a single outlier signal from dominating the classification.

**Thresholds:**

```
0.00 ──────────────────────────────────────────── 1.00
      [   HUMAN   ]   [   UNCERTAIN   ]   [   AI   ]
      0.00     0.35   0.35         0.65   0.65    1.00
```

The thresholds are asymmetric by design: a false positive (labeling a human as AI) is worse than a false negative on a creative platform. A score of 0.5 always falls in the uncertain zone and never triggers an AI label. To get `likely_ai`, both signals need to point clearly in the same direction, producing a combined score above 0.65.

**How I validated scores are meaningful:**

Ran the four test inputs from the assignment spec. Expected and actual results:

| Input | Expected | `llm_score` | `stylo_score` | `confidence` | `attribution` |
|---|---|---|---|---|---|
| Formulaic AI paragraph ("In conclusion…") | > 0.65 | 0.90 | 0.41 | 0.705 | likely_ai ✅ |
| Clearly AI paragraph ("paradigm shift…") | > 0.65 | 0.80 | 0.27 | 0.587 | uncertain |

## Transparency Label

All three variants are returned verbatim by `labeler.py`. The `{confidence_pct}` placeholder is computed at runtime.

**High-confidence AI** (confidence > 0.65):

```
⚠️ Likely AI-generated

Our system detected strong indicators that this content may have been produced
with an AI writing tool. Confidence: {confidence_pct}% AI.

This label reflects automated analysis and may not be accurate. If you wrote
this yourself, you can submit an appeal below — we'll review it and update
this classification.
```

**Authorship unclear / Uncertain** (confidence 0.35 – 0.65):

```
🔍 Authorship unclear

We weren't able to confidently determine whether this content was written by
a human or generated by an AI tool. Our signals disagreed or were too weak
to make a reliable call.

No action will be taken based on this result. You can still submit an appeal
if you'd like to add context about how this was written.
```

**Appears human-written** (confidence < 0.35):

```
✅ Appears human-written

Our analysis found no strong indicators of AI generation in this content.
Confidence: {confidence_pct}% human-authored.
```

**Design notes:** The AI label explicitly surfaces the appeal path because that's the route to correct a false positive — it must be visible at the moment the creator sees the label. The uncertain label says "no action will be taken" because a platform should not penalize a creator for an inconclusive verdict. The human label is brief — there is nothing the reader needs to do.

## Rate Limiting

**Limits applied to `POST /submit`:** `10 per minute; 100 per day`

**Reasoning:**

- **10 per minute:** A human creator submitting their own work would rarely submit more than 1–2 pieces in a minute. Ten is generous for legitimate use (batch testing, retries after edits) while blocking a script that tries to flood the system or probe the classifier systematically. At 10/min a determined adversary can still send 600 requests/hour, but the daily cap closes that window.
- **100 per day:** A prolific writer publishing daily might submit 5–10 pieces a day. 100 gives 10× headroom for legitimate power users while making it economically expensive to run a large-scale probing campaign (the Groq API has its own rate limits too, so per-request cost compounds).

These limits are in-memory (resets on server restart), which is appropriate for local development. A production deployment would use Redis-backed storage so limits persist across restarts and scale across instances.

## Stretch Features

### S1 — Ensemble Detection (3rd signal + voting)

**What was built:** A third independent detection signal (`detector_bigram.py`) and an updated ensemble scoring system in `scorer.py` that replaces the two-signal weighted average.

**Third signal — Bigram repetition rate (pure Python)**

Measures how often word pairs (bigrams) repeat across the text. AI writing reuses specific two-word combinations from its training distribution ("it is important," "in the context of," "as well as"). Human writing, especially creative writing, uses more varied collocations.

```
bigrams = all consecutive word pairs in the text (lowercased)
repetition_rate = (total_bigrams - unique_bigrams) / total_bigrams
bigram_score = min(1.0, repetition_rate / 0.08)   # linear: 0.08+ repetition → score 1.0
```

This signal is genuinely independent: Signal 1 is semantic, Signal 2 is sentence-level structural, Signal 3 is lexical/collocational.

**Updated ensemble weights:**
```
confidence = (0.45 × llm_score) + (0.30 × stylo_score) + (0.25 × bigram_score)
```

**Voting layer (attribution only, not used for confidence score):**

Each signal votes AI if its score > 0.50:
- 3/3 AI votes → `likely_ai`
- 2/3 AI votes + confidence > 0.65 → `likely_ai`
- 2/3 AI votes + confidence ≤ 0.65 → `uncertain`
- 0–1/3 AI votes → `likely_human`

The voting layer prevents a single outlier signal from dominating the attribution. The `POST /submit` response now includes `bigram_score`, `votes_ai`, and `bigram_detail` (repetition rate, total/unique bigram counts).

**Calibration note:** The bigram normalization ceiling is set at `0.08` (not the naive `0.30`) because within a single paragraph, bigrams rarely repeat more than 8% even in AI text — 30% repetition only occurs in much longer documents. Calibrated against paragraph-length test inputs.

**Verification results:**

| Input | `llm_score` | `stylo_score` | `bigram_score` | `votes_ai` | `confidence` | `attribution` |
|---|---|---|---|---|---|---|
| Formulaic AI paragraph | 0.90 | 0.41 | 0.58 | 2/3 | 0.673 | `likely_ai` |
| Casual ramen review | 0.00 | 0.24 | 0.00 | 0/3 | 0.073 | `likely_human` |

### S2 — Provenance Certificate

**What was built:** A `POST /verify` endpoint and `certificates.py` module that lets a creator earn a "verified human" credential on their content.

**How it works:**

The creator calls `POST /verify` with their `content_id` and a `verification_sample` — a second piece of their own writing not previously submitted. The system runs the full 3-signal pipeline on the sample and then applies three checks:

1. Original submission confidence < 0.40 (well into human territory)
2. Verification sample confidence < 0.40
3. Stylometric similarity — TTR and sentence length variance between the two texts must be within ±0.15 (same-author check)

If all three pass, a certificate is issued and stored against the `content_id`. Subsequent `POST /submit` responses for that creator include the certificate field.

**Certificate display in response:**
```json
{
  "certificate_issued": true,
  "content_id": "230acd41-6ffc-42e4-84d2-2f7892a120ec",
  "display_label": "✦ Verified Human — Creator has passed an additional verification step.",
  "reasoning": "Stylometric profiles are similar (TTR diff: 0.054, variance diff: 0.148)"
}
```

**Failure response** (returns HTTP 422):
```json
{
  "certificate_issued": false,
  "content_id": "...",
  "display_label": null,
  "reasoning": "Verification sample confidence too high (0.671 ≥ 0.40 threshold)."
}
```

**Endpoint:** `POST /verify` — `{ content_id, verification_sample, original_text? }`

**What was built:** A `GET /analytics` endpoint backed by `analytics.py` that reads `logs/audit.jsonl` and computes three metrics across all submissions.

**Three metrics:**

1. **Detection distribution** — count and % of submissions in each attribution category (`likely_ai`, `uncertain`, `likely_human`)
2. **Appeal rate** — total appeals as a % of total submissions; a high rate signals potential miscalibration

### S4 — Multi-Modal Support

**What was built:** Extended `POST /submit` to handle a second content type — `image_description` — in addition to plain `text`. Classifies whether an image caption was written by a human or generated by an AI captioning tool.

**How it works:**

The request body accepts an optional `content_type` field (`"text"` or `"image_description"`, defaults to `"text"`). When `image_description` is passed:

- **Signal 1 (LLM)** uses a modified system prompt tuned for image descriptions, describing the difference between AI alt text ("A serene mountain landscape with snow-capped peaks reflecting in a crystal-clear lake") vs. human captions ("dad at the beach, he hated that hat").
- **Signal 2 (stylometrics)** swaps out `punctuation_density` (not meaningful for captions) for an **SVO pattern score** — counts `"A [noun] [verb]ing in/on/with a [noun]"` matches, which AI captioning tools overuse.
- **Signal 3 (bigrams)** applies unchanged.
- **Thresholds recalibrated** — image descriptions are short and produce noisier scores, so the uncertain zone widens: `likely_human` < 0.30, `uncertain` 0.30–0.70, `likely_ai` > 0.70.

The response includes `content_type` so the caller knows which pipeline ran. The audit log records `content_type` on every entry.

**Verification results:**

| Input | `llm_score` | `stylo_score` | `bigram_score` | `votes_ai` | `confidence` | `attribution` |
|---|---|---|---|---|---|---|
| AI landscape caption | 1.0 | 0.20 | 0.00 | 1/3 | 0.51 | `uncertain` |
| "dad at the beach, he hated that hat" | 0.0 | 0.20 | 0.00 | 0/3 | 0.06 | `likely_human` |

The AI landscape correctly avoids a false `likely_human` — the widened `uncertain` zone (0.30–0.70) catches it despite only the LLM voting AI. Stylometric and bigram signals can't work on single-sentence captions, which is why the conservative thresholds exist. The voting logic falls back to confidence-based thresholds when votes ≤ 1, so `uncertain` is returned instead of overriding to `likely_human`.

## Known Limitations

### High-quality narrative AI writing (false negative)

The system will misclassify well-crafted literary AI output as human-written.

**Example that triggered this:**
> "The lighthouse keeper climbed the iron stairs for the thousandth time that season, his weathered boots echoing against cold metal. The beam swept across the dark ocean with mechanical precision, casting his shadow against whitewashed walls. No ships came anymore—fewer each year—but he tended the light as faithfully as he tended his sparse garden. At night, he sat with black coffee and older memories, watching the beam pierce the fog, wondering if anyone was still watching back."

Actual scores: `llm_score: 0.00, stylo_score: 0.40, confidence: 0.159` → `likely_human`. This is a confirmed false negative.

**Why both signals fail:**

- **LLM signal:** The text contains none of the surface markers the classifier associates with AI output — no hedging, no over-balanced structure, no formulaic transitions. The LLM reads atmospheric prose with a consistent narrative voice and scores it as human.
- **Stylometric signal:** Sentence lengths vary (14, 13, 10, 25 words), vocabulary is diverse and evocative (high TTR), and punctuation is restrained. All three sub-metrics point human. The signal has no mechanism to detect *collocational* AI patterns — phrases like "mechanical precision," "casting his shadow," "as faithfully as" that recur across AI-generated literary text.

**Root cause:** Both signals measure surface uniformity. High-quality AI writing is designed to avoid surface uniformity. Neither signal operates at the level where literary AI is detectable: collocational frequency across a corpus of AI-generated prose.

**What would help:** A bigram/collocation repetition signal compared against a reference corpus of known AI output (planned as Stretch Feature S1). This is a silent false negative — a creator receiving `likely_human` on AI-generated work would not be incentivized to appeal, which is the more benign failure mode on a creative platform (a false positive is worse).

## Spec Reflection

**One way the spec helped:** Writing the three transparency label variants verbatim in `planning.md` before touching `labeler.py` made the implementation a direct transcription rather than a design problem. The exact strings were already decided; the only implementation question was how to substitute `{confidence_pct}`. Without that pre-decision, it would have been easy to write labels that were technically functional but unclear to a non-technical reader.

**One way implementation diverged:** The spec called for `storage.py` to use either in-memory or SQLite. In implementation, in-memory was chosen because it's simpler and the assignment doesn't require persistence across restarts. The consequence is that `POST /appeal` only works for `content_id` values from the current server session — if the server restarts, all stored records are lost and existing IDs return 404. This is documented behavior, not a bug, but it's a gap between spec intent (a persistent store) and what was built.
